# 飞猪房型数据获取 - 方法对比与推荐

## 一、你当前面临的问题

- **现象**：在飞猪 App 酒店房型列表页，用 Appium Inspector 抓到的页面源里，**没有可用的文案节点**（房型名、价格等全是 `text=""`），解析结果一直是 0 条。
- **WebView**：点「地球图标」切换环境时提示 **「没有检测到其他环境」**，即没有 WebView 可切，无法从网页 DOM 拿数据。
- **结论**：在你这台设备 + 当前飞猪版本下，**既不能靠原生 UI 树，也不能靠 WebView 取 HTML**，只能从「数据来源」另找办法。

---

## 二、各方法优缺点

| 方法 | 优点 | 缺点 | 你当前是否可用 |
|------|------|------|----------------|
| **1. UI 树解析**（Appium/ u2 取 XML → parse_feizhu_xml） | 不碰网络、不装证书；和美团一样流程统一。 | **依赖页面暴露文案**。飞猪该页未暴露 → 你这边用不了。 | ❌ 不可用（无文案节点） |
| **2. 抓包 + 接口 JSON**（Charles/Fiddler 抓响应 → feizhu_api_to_json） | **数据最准、最全**；和 App 显示一致；一次抓多次用；不依赖 UI/WebView。 | 要配代理、装证书；需能找到房型列表接口。 | ✅ 可用 |
| **3. WebView 取 HTML**（切到 WEBVIEW 取 Page Source → parse_feizhu_html） | 不抓包；若 H5 结构稳定，解析后可复用。 | **依赖存在 WebView 且能切 context**。你这边「没有检测到其他环境」→ 用不了。 | ❌ 不可用（无 WebView 环境） |
| **4. 截图 + OCR** | 不抓包、不依赖 UI 树和 WebView；任何能截到的都能试。 | 准确率受排版/分辨率影响；多屏要多次截；维护成本高。 | ✅ 可用（但不推荐为首选） |

---

## 三、推荐方法：抓包 + 接口 JSON

**推荐你用「抓包拿到房型列表接口的响应 JSON，再用 feizhu_api_to_json.py 转成 1.json」。**

### 理由

1. **和你当前情况匹配**  
   UI 无文案、又无 WebView 可切，数据只能来自「App 自己请求的接口」。抓包是唯一能稳定拿到和界面一致、且结构化的数据的方式。

2. **数据质量最好**  
   接口返回的是结构化 JSON（房型名、价格、早餐、可订状态等），字段清晰，不需要猜布局；比 OCR 稳定、比解析不存在的 UI 节点靠谱。

3. **一次配置，长期可用**  
   配好代理和证书后，只要飞猪接口不变，同一套抓包 + 脚本可反复用；换酒店、换日期只需重新操作 App 触发请求即可。

4. **项目里已就绪**  
   已有 **Feizhu/抓包接口方案.md** 和 **Feizhu/feizhu_api_to_json.py**，你按文档抓一次、保存 JSON，就能跑通并得到和现有 1.json 同格式的结果。

### 不首选 OCR 的原因

OCR 作为「实在不能抓包」时的备选可以，但：  
要自己处理多屏、排版变化、识别错误，后续维护成本高，且准确率不如接口数据，所以不作为首选。

---

## 四、小结

| 项目 | 说明 |
|------|------|
| **你面临的问题** | 飞猪房型页 UI 无文案、无 WebView 环境，无法用 UI 解析和 WebView HTML。 |
| **推荐方法** | **抓包 + 接口 JSON**，用 `feizhu_api_to_json.py` 转成 `Feizhu/1.json`。 |
| **原因** | 唯一稳定、结构化且与界面一致的数据来源；脚本现成，一次配置可复用。 |

具体步骤见 **Feizhu/抓包接口方案.md**；若抓到的接口字段和脚本里预设的不一致，把一份响应样本（可打码敏感信息）发出来，可以按实际字段改解析逻辑。
